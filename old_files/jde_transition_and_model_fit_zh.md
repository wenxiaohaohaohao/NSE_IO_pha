# 面向 JDE 的两件关键事：如何做转移矩阵、当前模型够不够

## 1. 先说结论

如果你的目标是把文章做得更像 `Journal of Development Economics` 会接受的版本，那么当前最优组合确实不是先冲完整结构模型，而是：

1. 用 `DID / event study` 把创新效应做扎实
2. 用转移矩阵把组织重构做出来
3. 用委托生产 / 持证 / CMO / CDMO 证据补强商业化机制
4. 最后用一个中等复杂度的行业模型解释

在这个组合里，**转移矩阵是最值得优先完成的机制证据**，因为它直接回答：

- 企业组织形式有没有变化
- 变化发生在哪些类型之间
- 变化主要来自进入，还是来自 incumbents 自身的重构

## 2. 转移矩阵到底在做什么

转移矩阵不是一个复杂模型，它本质上只是在回答：

> 一家企业今年属于哪一类，明年变成了哪一类？

你当前最合适的四类分类是：

- `研发型`
- `生产型`
- `混合型`
- `转型期`

于是转移矩阵就是一个表：

- 行：企业在 `t` 年属于哪一类
- 列：企业在 `t+1` 年属于哪一类

比如：

| from \ to | 研发型 | 生产型 | 混合型 | 转型期 | exit |
| --------- | ------ | ------ | ------ | ------ | ---- |
| 研发型    | ?      | ?      | ?      | ?      | ?    |
| 生产型    | ?      | ?      | ?      | ?      | ?    |
| 混合型    | ?      | ?      | ?      | ?      | ?    |
| 转型期    | ?      | ?      | ?      | ?      | ?    |

如果你把每一行标准化成和为 1，这个表就变成了：

- 某类企业一年后留在本类的概率
- 跳到别的类的概率
- 退出的概率

这就是“组织重构”的最直接证据。

## 3. 为什么转移矩阵对你的文章这么重要

因为你现在最大的风险是：

- 只有创新回归
- 但没有机制上的组织事实

如果你只有“MAH 提高创新”，审稿人会问：

- 为什么提高？
- 是因为研发更多？
- 是因为审批更快？
- 是因为有更多研发型进入者？
- 是因为生产型企业转型了？

转移矩阵的作用就是告诉读者：

- 企业组织状态真的变了
- 变化方向与 MAH 的制度逻辑一致

例如，如果你发现政策后：

- `混合型 -> 研发型` 上升
- `生产型 -> 混合型` 上升
- `研发型` 新进入者增多

那你的故事就会很强：

> MAH 并不只是让原有企业多做一点创新，而是改变了行业组织结构。

## 4. 转移矩阵具体怎么做

### 第一步：先把分类规则固定

最重要的不是先跑表，而是先把分类规则固定。

你现在的规则已经有基础了，比如：

- 技术人员占比高 -> `研发型`
- 生产人员占比高 -> `生产型`
- 两者接近 -> `混合型`
- 某些异常组合 -> `转型期`

关键要求是：

1. **规则要稳定**
2. **政策前后用同一套规则**
3. **不要为了让结果好看而改规则**

否则转移矩阵就没有可信度。

### 第二步：做 firm-year 面板

你需要一张最基础的 firm-year 数据：

- `firm_id`
- `year`
- `type4`

其中 `type4` 是四类企业标签。

只要这三列稳定，你就已经可以开始做转移矩阵。

### 第三步：定义转移

对每一家企业 `i`：

- 看它在 `t` 年的类型
- 看它在 `t+1` 年的类型

这就是一条转移记录：

- `mix -> rd`
- `prod -> mix`
- `rd -> exit`

等等。

### 第四步：定义 exit

如果企业在 `t` 年存在，但在 `t+1` 年不存在，而样本并没有结束，就记为：

- `type_t -> exit`

这一列非常重要。

因为 MAH 可能不仅影响“变成谁”，还影响“会不会消失”。

### 第五步：把 entry 单独拿出来

这一点非常重要：

**entry 不能和 incumbent switching 混在一起。**

如果一个企业第一次出现在样本里，你要单独记：

- `entry -> rd`
- `entry -> prod`
- `entry -> mix`
- `entry -> transition`

这样你才能回答两个不同问题：

1. 原有企业有没有重组？
2. 新进入者主要是什么类型？

这两个机制在经济学上完全不同。

### 第六步：按政策前后分别做矩阵

你最少要做两张：

- pre-policy matrix
- post-policy matrix

推荐做法：

- pre-policy origin years: `2011-2014`
- post-policy origin years: `2016-2022`

这样避免 `2015 -> 2016` 跨制度阶段污染 pre matrix。

### 第七步：先做 counts，再做 shares

你需要两种版本：

1. **计数矩阵**
   - 多少家企业发生了这种转移
2. **行标准化矩阵**
   - 某类型企业转向别类的概率是多少

在论文里，行标准化矩阵更有解释力。

## 5. 转移矩阵做完后你最该看什么

不要把所有数字都同等对待。

最该盯住的是这些方向：

- `混合型 -> 研发型`
- `混合型 -> 生产型`
- `生产型 -> 混合型`
- `研发型 -> 混合型`
- 各类型的 `exit`
- 新进入者分布

因为这些最直接对应你的模型：

- `A -> B`
- `A -> C`
- `C -> A`
- `B -> A`

## 6. 委托生产 / 持证 / CMO / CDMO 证据怎么补

这部分你不一定一开始就有很完美的量化变量。

最现实的路线是分两层。

### 第一层：文本与制度证据

去找：

- 上市公司年报
- 招股说明书
- 企业公告

检索关键词：

- MAH
- 上市许可持有人
- 持证
- 委托生产
- 受托生产
- 委托加工
- CMO
- CDMO

你要看的不是一句话，而是：

- 政策后是否更多企业提到这些安排
- 哪类企业更常提
- 这些表述是否与组织分类一致

### 第二层：财务或经营代理变量

如果有更好的财务数据，可以看：

- 营收增长
- 毛利率
- 营业利润率
- 销售费用率
- 是否出现制造服务收入

目标不是完美识别，而是给读者一个合理印象：

> `C` 型企业确实因为 outsourcing margin 的出现而变得更重要。

## 7. 你现在的模型够不够

### 7.1 方向上：够了

你现在的模型方向是对的，因为它已经有：

- 三类组织状态 `A/B/C`
- idea generation 与 implementation 的分离
- 进入、退出、状态转移
- 一个行业内部市场清算
- welfare / GE 讨论

这已经比单纯的 PE 创新故事强很多。

### 7.2 对 JDE 来说：还要“收”

你现在的模型**方向对，但还需要收敛成一个更可执行的版本**。

对 JDE 来说，最好的模型不是最大，而是最有纪律。

你当前模型最容易过度扩张的地方有三个：

1. `implementation`
2. `融资支持`
3. `组织状态转移`

这三者都该保留，但要分主次。

### 7.3 最推荐的主结构

我建议你最终的中等复杂度模型只保留这四个核心块：

1. 三类企业：`A/B/C`
2. `B` 的 idea generation 与 implementation 选择
3. `A/B/C` 之间的状态转移
4. `B` 与 `C` 之间的委托生产市场清算

这样就够了。

### 7.4 融资支持怎么处理最合适

融资支持可以保留，但建议作为：

- `entry margin` 的 reduced-form 强化项

而不是单独再建一套融资合同或 VC 模型。

也就是说：

- 主机制：商业化摩擦下降
- 次级放大器：创新型进入与融资支持增强

这是最稳的写法。

## 8. 所以“模型够吗”的最终判断

如果你现在问：

> 这个模型能不能作为 JDE 风格文章的理论部分？

我的回答是：

**够，但前提是你把它压成“中等复杂度版本”，不要继续往并购、融资合同或复杂多产品方向扩。**

更具体地说：

### 够的部分

- `A/B/C` 结构
- implementation 机制
- 状态转移
- 行业均衡

### 还需要实证来支撑的部分

- 组织重构事实
- 委托生产 / 持证 / CMO / CDMO 证据
- 创新型进入

### 不建议现在硬做的部分

- 完整结构估计
- 完整 VC 合同
- 复杂 bargaining
- 大型多产品质量阶梯模型

## 9. 我对你的最实用建议

你下一步最值得优先做的顺序是：

1. 先把四类分类规则锁定
2. 先做两张转移矩阵
3. 把 entry 单独拆出来
4. 补委托生产 / 持证 / CMO / CDMO 证据
5. 再回头收模型

不要反过来：

- 先把模型越写越大
- 再去找数据硬塞

对你这篇文章来说，**事实应该先行，模型要跟着事实走。**

## 10. 如果转型低频，论文里应该怎么解释

如果最后的数据呈现出这样一种模式：

- 大多数企业长期停留在原有类型；
- 少数企业在某几年发生一次关键转型；
- 转型完成后又在新类型下稳定下来；

这并不是坏结果，反而很可能是最符合经济学逻辑的结果。你可以在论文里用下面这段正式中文来解释：

> 企业组织形式具有明显的惯性，组织重构并非年度经营中的连续微调，而是伴随人员结构、资质安排、生产协同与管理边界调整的高固定成本离散决策。因此，政策冲击并不必然表现为所有企业在每一期频繁切换类型，更可能表现为一部分边际企业在政策后完成一次性重组，并在新的组织状态下保持稳定经营。就本文而言，如果转移矩阵显示多数企业长期停留在原类型、少数企业在若干年内发生一次关键转型后趋于稳定，这并不削弱 `MAH` 的组织重构机制，反而与“`MAH` 降低重组门槛、触发边际企业跨越组织转换门槛”的制度逻辑一致。因而，本文更关注政策是否提高关键转移方向的发生概率和累积重组比例，而不预期所有类型之间都出现高频、双向且持续的年度流动。

这段话有三个作用：

- 它把“低频转型”解释为**高调整成本 + 组织惯性**的自然结果；
- 它把政策作用理解为**改变边际企业的重组时点**，而不是让所有企业持续高频波动；
- 它为后面把实证从简单转移矩阵扩展到 `hazard` 和 `cumulative transition` 提前铺垫。

## 11. 如果转型低频，实证应该怎么改做

如果你发现年度转移矩阵的对角线非常大、非对角元素很小，不要立刻否定组织重构机制。更合适的做法是把“年度流动”改写成“关键转型的发生时点”和“政策后的累积重组”。

### 11.1 年度转移矩阵仍然要保留

年度转移矩阵仍然是必要的，因为它至少告诉读者两件事：

- 企业组织状态具有很强的持久性；
- 重组确实是低频离散事件，而不是每年小幅摆动。

因此，年度转移矩阵应继续作为**描述性事实**保留在正文或附录里，但它不应成为检验机制的唯一统计对象。

### 11.2 把“是否发生第一次关键转型”改写为 hazard

如果你更关心 `混合型 -> 研发型`、`生产型 -> 混合型` 这类关键方向，可以把它们定义为“首次转型事件”。

做法是：

1. 先选定一个起始状态，例如所有在 `t` 年属于 `混合型` 的企业；
2. 只保留仍处于该起始状态、尚未发生目标转型的企业作为 risk set；
3. 定义

\[
Hazard_{i,t}^{s \to j}=1
\]

当且仅当企业 `i` 在 `t` 年仍为 `s`，并在 `t+1` 年第一次转为 `j`；
4. 在这个 risk set 上估计政策是否提高了该首次转型事件的发生概率。

直观地说，`hazard` 回答的是：

> 对于那些“还没有重组、但有可能重组”的企业，`MAH` 有没有提高它们在这一年完成关键转型的概率？

这比直接盯着整张年度转移矩阵更适合低频组织重构。

### 11.3 离散时间 hazard 的经验写法

你不必一开始就做复杂的连续时间生存模型。对目前这篇文章，更实用的是离散时间 hazard 写法，例如在线性概率模型或 logit 框架下估计：

\[
\Pr\left(Hazard_{i,t}^{s \to j}=1 \mid i \text{ at risk in } t\right)
=
F\left(\alpha + \beta \cdot Policy_{i,t} + \lambda_t + X_{i,t}'\gamma\right),
\]

其中：

- `Policy_{i,t}` 可以是试点暴露、政策后指标，或两者交互；
- `\lambda_t` 控制年份共同冲击；
- `X_{i,t}` 可以加入企业规模、年龄、所有制、地区等控制变量。

如果你的制度冲击有试点和全国推广两个阶段，这个 hazard 设计尤其有用，因为它能把“重组何时发生”而不是“平均一年流动多少”作为识别重点。

### 11.4 再做 cumulative transition

低频转型还有一个非常适合的统计对象：**累积转型**。

定义方式很简单。对每一家在基期属于 `s` 类型的企业，定义：

\[
CumSwitch_{i,[t_0,t_1]}^{s \to j}=1
\]

当且仅当企业在区间 `[t_0,t_1]` 内任意一年曾经从 `s` 转到 `j`。

这样做的优点是：

- 它允许企业“几年后才转”，不会因为一年内没动就把政策效果漏掉；
- 它非常适合“转一次、然后稳定”的组织重构；
- 它能更自然地对应你模型里的高转换成本和门槛跨越。

经验上，你可以分别报告：

- 政策前窗口内的累积转型率；
- 政策后窗口内的累积转型率；
- 关键转型方向的差异；
- 不同起始类型的异质性结果。

### 11.5 一套更稳的证据链

如果转型低频，我建议你把证据链改成下面这套：

1. 用年度转移矩阵展示“高惯性、低频重组”的基本事实；
2. 用 `hazard` 结果展示 `MAH` 是否提高关键首次转型的发生概率；
3. 用 `cumulative transition` 展示政策后若干年内累积重组是否更强；
4. 再把这些与创新结果、委托生产 / 持证 / `CMO` / `CDMO` 证据放在一起解释。

这样你就不是在硬说“企业每年都在变”，而是在更准确地说：

> `MAH` 改变的是边际企业完成组织重构的时点和概率，而不是让整个行业陷入高频流动。

### 11.6 对模型的反向约束

如果数据最终确实显示“低频、一次性、稳定化”的重组模式，这对模型来说反而是一个支持，而不是麻烦。因为它说明：

- 状态转移应被理解为**有固定成本的离散调整**；
- `\kappa_{ij}` 这类转换成本是有经验基础的；
- 你不应该把模型写成企业每年都在 `A/B/C` 之间高频来回切换。

换句话说，低频转型不会削弱你的模型，反而会帮助你把模型收敛到一个更符合 `JDE` 风格的中等复杂度版本。
